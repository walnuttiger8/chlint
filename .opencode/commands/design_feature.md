# Проектирование фичи. Схемы + план кода

Ты - профессиональный архитектор ПО который продумывает архитектуру для фичи.
Фича - задача, изменение или нововведение в проекте которое необходимо реализовать.

**Основной принцип:** Определи ЧТО и ПОЧЕМУ перед КАК. Не планируй код до согласования архитектуры.

---

## Фаза 0: Пойми задачу

### Фаза 0.1: Парсинг аргументов

- `$1` - название фичи (slug, используется для названия директорий)
- `$2` - описание задачи, требования или файл с описанием. Все что идет после названия

Если аргументов не хватает, спроси:

```
Пожалуйста укажите:
1. Название фичи (slug, пример: new-auth)
2. Описание фичи
```

Если название фичи уже есть, попроси пользователя описать фичу перед тем как продолжить.

### Фаза 0.2: Ознакомление с задачей

- Прочитай описание задачи от пользователя.
- Определи **основную цель** задачи.
- Определи критерии приемки.

В конце этой фазы задай вопрос пользователю:

```
Праивльно понимаю, что цель задачи: {основая цель}?
```

Переходи к следующим шагам только после получения явного подтверждения.

### Фаза 0.3: Ознакомление со стандартами проекта

После того как понял задачу и согласовал **основную цель** с пользователем.

Прочитай ВСЕ стандарты проекта:
- прочитай ВСЕ файлы из директории `/docs/conventions`


## Фаза 1. Проектирование

Спроектируй архитектуру фичи. 
Используй несколько файлов для отображения разных аспектов архитектуры.

Предпочитай схемы на mermaidjs для визуализации.

Создай отдельные файлы для:
- описания модулей

Расположи все файлы в директории `/docs/features/{feature_name}` где feature_name это название фичи 
из аргументов пользователя.

После завершения этой фазы, запроси ревью от пользователя. 
НЕ переходи к следующей фазе пока без явного разрешения.

## Фаза 2. Планирование

После согласования архитектуры, приступай к формированию детального плана реализации в виде **нескольких файлов на фазу** в директории `plan/`.

### Выбор стратегии порядка фаз

Определи порядок фаз на основе задачи

**Вариант 1: Вертикальное разделение.** 
Если изменения затрагивают несколько модулей, разбей каждый модуль на отдельные фазы.

**Вариант 2: Горизонтальное разделение.** 
Изменения по слоям (domain -> application -> infrastructure).

Обоснуй выбор стратегии в `plan/README.md`

### Структура вывода

```
/docs/features/{feature_name}/plan
|-- README.md
|-- phase-01.md
|-- phase-02.md
|-- phase-NN.md
```

### Зачем несколько файлов?

- Каждая фаза **самодостаточна** и независима. Исполнитель читает только ОДИН файл
- Каждая фаза проходит ревью и согласовывается независимо
- Руководитель может предоставить файл агенту-исполнителю без посторонних шумов
- Отслеживание прогресса: в README.md. Файл фазы остается для справки.
